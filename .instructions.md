# Atlas Project - Code Patterns & Standards

All AI tools and developers must follow these strict patterns when working on this project.

## Routes

### Lazy Loading
All routes should use Angular lazy-loading. This involves using `loadComponent` instead of the `component` property when registering a route.

### Route Data Availability
If a route requires data to be available, then naturally it must use a guard or a resolver to provide that data before the route is used. If the data is static, then you can and should use the `data` property of the route. An example is a page needed for a specific route.

### Error Routes
Instead of staying on the same page when data that is required is unavailable to that route, redirect to an error-displaying route page. E.g., the page a route needs can't be found so the user should be redirected (from within the resolver) to the error page.

## Components

### Isolated Component Responsibility
One intended side effect of components is their ability to be copied to entirely different projects and not losing almost any functionality (as in, the entire component is almost self-surviving).

For example, a modal component should sub-divide all of its responsibilities within itself - modal header, modal body, modal footer. 

Further, the management of the modals show, hide, and user functionalities should be self-contained and not rely on outside userland code. E.g., closing the modal should be entirely handled by the modal's own code. 

Modals are an example, but this self-contained responsibility goes for any component you make. Even inputs, sublabels for inputs, regular labels, and so on.

### Layout Components
Do not copy-paste and duplicate the same code to structure a page component again and again when you should be using parent-child components in routes to handle duplicate layout design. For example, an interior page should have an interior parent component that implements all duplicate parts of the layout separate from the individual page (sidebar, header, nav, footer, and so on).

### Dynamic Components
Components that spawn other components should always be dynamically rendered in code using templates and containers. We don't want to clog up the DOM but we also want to be able to control these dynamic components in our TypeScript and not with HTML Angular-template language control blocks (if statements, for example).

An example would be dropdown menus or spawning a modal. Wrap the part of your HTML that needs to be dynamically rendered in an `<ng-template>` and have an `<ng-container>` available for that template to be created into. Google "Angular create embedded template into container" for code guides on how to do this.

If you need to have the template be spawned into a location that is outside of all elements (such as in the document root) then you will need to add a "ViewContainerRef" to your App component and your descendant component (e.g. dropdown or modal) will inject an "AppRef" so you can get the view container reference of the application to spawn the component into. This means you won't use an `<ng-container>` in your own component.

### Reusable Code - Future Proofing
When you write code for a page, don't think about the code being just for that page as your primary thought. First consider: "Would we ever use this on another page?" and after asking yourself this, decide if you should componentize and/or service-ize that code so it can be used consistently in other places.

Examples of this are notifications, modals, dropdowns, inputs, cards, and alert boxes; but this applies to all code you write. Avoid mass duplication of work and help your future self - or other developers - reuse your code for consistency across the application and for ease of use. It helps nobody if they have to remake a dropdown for every page or recreate an alert/notification system that no longer is consistent on every page.

### Duplicate Components
Before creating a component, review the codebase's components (almost always found inside "app/components" or in that project's domain-split directory's components folder) for components you can use or possibly modulate before deciding to create a new one.

If you're working with a mature codebase and need a dropdown, chances are that component might already exist.

### Mobile-Friendly Components
Unless absolutely necessary, do not use TypeScript or Angular code to mobile-ify a component. Try to always use pure CSS (media queries, container queries, etc). Code that tries to read the window size should be exceptionally rare and only when every other possibility has been exhausted.

### Components Needing Information About External Components
When a component needs information about a component that is not a child of itself (imagine needing the nav sidebar's open or close state being needed) then you should opt for creating a service for the data you need.

In the example that you need the open or close state of the navigation sidebar from a component that isn't related to the sidebar, the sidebar should have a service. Something like "nav-sidebar-service". There should be a signal for IsOpen in that service. Then both the sidebar and the unrelated component should be reading/using that IsOpen signal in the service - and no more "IsOpen" signals should be in the sidebar itself. This way both components can inject the sidebar service and read from the shared signal.

## Angular-Code-Specific Patterns

### Signals for All Data Used in HTML Templates
If any variable, property, data, etc. is to be used in the HTML template files, then it must be a signal of some type. Signals aren't just created with `signal()` but also with `input()`, `viewChild`, `contentChild`, `viewChildren`, and so on.

This does not apply to static data such as references to enums.

### Remove Square-Bracket Attributes for Simple Data Binding
Two examples below:

```html
<component [Data]="'Hello'">
```

…and

```html
<component Data="Hello">
```

When you don't need Angular to evaluate the expression/code inside the attribute in your HTML template, use the simpler data binding attribute format (without square brackets). Usually, this applies to string value bindings only.

### Naming Outputs
Prefer naming outputs with the "On" prefix. Such as "OnDeleteRequested" instead of "DeleteRequested"

## Mirroring Backend Data

### Enums
If the backend has an enum also used on the frontend (Say, in an API call) then you should mirror it. As in, copy it and replicate it as a frontend enum. Do not use the backend to query for enum cases.

## API Services

### Organization
Prefer to use a single folder for any services that interact with backend APIs. Our projects typically use "api-services" in the app directory, but if the application is split in domain usage (like the ClientDashboard's client/staff ends) then each of those domain folders can have their own api-services directory.

### Interfaces for Body Payloads and Parameters
For the API calls that require inputs in some form (such as URL parameters or payload bodies), you should opt to create a directory in your "api-services" folder called "interfaces" and store them in the name format of "ISomethingBody.ts" (for example: ICreatePageBody.ts)

However, when you have a common parameter (such as query parameters for List-style API calls to fetch objects) list which may have duplicate parameters (think: page, limit, query, sortName, sortDirection) then these should go in the applications global interfaces directory (i.e., app/interfaces) in a sub-directory called "api".

An example is if you have ListObjects (ListPages, ListUsers, and so on) then you would create an interface in app/interfaces/api/IListObjectsRequest.ts which had parameters:

- `string? Query`
- `int Page`
- `int Limit`
- `string? SortColumnName`
- `int SortDirection`
